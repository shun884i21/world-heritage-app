## 3. 一番大事な「次の仕事」＝ AI解説の続き

残り **約999件** の解説を継続する。方針：**著名どころから順に、12件前後の小バッチでマージ＆プッシュ**しながら積み上げる（1ターンで数バッチ）。ユーザー指示（「ヨーロッパを50件」等）が来たらその範囲を優先。

### 絶対のルール（ユーザーとの約束）

> **記憶で書かない。出典（Wikipedia本文）を取得して実際に読み、その記述だけで書く。** 記事に無い年号・人物は書かない。断定できない箇所は「〜とされる」等にトーンダウンする。各解説に出典URL＋取得日を必ず残す。

ハルシネーション防止が最優先。有名サイトはこの方式で十分高品質に書ける。

### 再現手順（このとおりやれば続けられる）

すべて `world-heritage-app\` をカレントに、`node`（v22）で実行。

1. **対象IDを特定**（`data/sites.json` を名前/国で検索）。`scripts/` に検索用の使い捨てスクリプトを作ってもよい。例: 国名 `s.countries.includes("フランス")`、名前 `s.name.includes("...")`。
   - ⚠️ 著名サイトでも `name` が英語のまま（Wikidataに日本語名なし）や `ja_wiki` が無いものがある。その場合は後述の対処。

2. **出典本文を取得**: `node scripts/fetch_wiki.mjs <id1> <id2> ...`
   - 各IDの `ja_wiki`（無ければ `en_wiki`）からMediaWiki APIで本文プレーンテキストを取得し `data/wiki_cache/<id>.json` に保存（冒頭3000字に制限）。
   - **Wikipediaリンクが無いID**は、タイトル直指定で取る必要がある（過去に佐渡=1698、アンコール=668、万里の長城=438、ガラパゴス=1、ギザ=86、ヴェルサイユ=83、クレムリン=545でやった）。汎用スクリプト **`node scripts/fetch_wiki_title.mjs <id> <ja|en> <タイトル>`** で `data/wiki_cache/<id>.json` を生成できる（フォーマットは他のキャッシュと同じ）。

3. **精読して執筆**: `data/wiki_cache/<id>.json` を Read で読み、各遺産につき以下3節を**本文の事実だけ**で書く。
   - `background`（歴史的背景, 4〜6文）/ `people`（関わった人々）/ `highlights`（見どころ）
   - さらに `source_url`（cacheのと同じ）, `source_label`（例「Wikipedia「姫路城」」）, `retrieved`（cacheのと同じ日付）。
   - 文体は既存52件に合わせる（です・ます調、観光客が読んで面白い温度感、登録基準は「登録基準(9)」のように軽く触れる程度）。
   - **自然遺産で「関わった人物」が薄い場合**は、保全活動・発見者・命名の由来などを `people` に書く（無理に人物を捏造しない）。

4. **バッチJSONにまとめてマージ**: 数件〜10件を1つの `batch.json`（`{ "<id>": {background,people,highlights,source_url,source_label,retrieved}, ... }`）として scratchpad等に書き、
   `node scripts/merge_descriptions.mjs <batch.jsonのパス>` で `data/descriptions.json` に統合（id昇順で保存、既存は上書き）。

5. **公開反映**: `git add -A && git commit && git push`（下記 [8. デプロイ](#8-デプロイ更新の反映)）。数分でライブ反映。

### 日本語名が無い著名サイトの扱い

`scripts/build_sites.mjs` の `NAME_OVERRIDE`（id→日本語名）に追記して `node scripts/build_sites.mjs` で `sites.json` を再生成する。現在: 438万里の長城 / 1ガラパゴス諸島 / 86ギザ / 83ヴェルサイユ / 162アミアン大聖堂（Wikidata名"ノートルダム大聖堂"は誤解を招くため上書き）/ 29クラクフ歴史地区 / 80モン・サン＝ミシェルとその湾 / 95ドゥブロヴニクの旧市街 / 97スプリトの史跡群とディオクレティアヌス宮殿 / 64ティカル国立公園 / 728エディンバラの旧市街と新市街 / 383セビリアの大聖堂・アルカサル・インディアス古文書館 / 541ヴィリニュス旧市街を登録済み。

## 4. データ構造

### `data/sites.json`（1171件の配列）
各要素: `id`(UNESCO参照番号), `name`(日本語、無ければ英語/上書き), `name_en`, `type`(cultural/natural/mixed/unknown), `criteria`(例"(i),(iv)"), `year`, `countries`[], `danger`(bool), `lat`, `lon`, `image`(Wikimedia Commons FilePath URL), `ja_wiki`, `en_wiki`, `qid`

### `data/descriptions.json`（id→解説、現在52件）
`{ "661": { background, people, highlights, source_url, source_label, retrieved } }`
アプリは起動時に読み込み、詳細モーダルで該当IDがあれば表示、無ければ「準備中」表示。

## 5. データ生成の元（再ビルド可能）

- `data/wikidata_raw.json`（5.5MB）= Wikidata SPARQL結果の生データ。`build_sites.mjs` の入力。
- `data/danger_raw.json` / `data/danger_refs.json` = 危機遺産リスト（Wikipedia「List of World Heritage in Danger」のCurrent listから抽出した参照番号）。
- `data/wiki_cache/<id>.json` = 各遺産のWikipedia本文キャッシュ（解説執筆の根拠）。
- `scripts/build_sites.mjs` … 生データ→sites.json（名寄せ・分類導出・危機遺産付与・名前上書き）
- `scripts/extract_danger.mjs` … 危機遺産参照番号の抽出
- `scripts/fetch_wiki.mjs` … Wikipedia本文取得
- `scripts/merge_descriptions.mjs` … バッチを descriptions.json に統合
- `scripts/make_icons.mjs` … アイコンPNG生成

## 6. アプリのファイル

- `index.html` … 3タブ＋詳細モーダルのDOM
- `css/style.css` … スタイル（緑×金の図鑑風テーマ）
- `js/app.js` … 全ロジック（フィルタ/検索/詳細/統計/コレクション/地図/SW登録）
  - 検索は フリーワード＋分類＋国/地域＋危機遺産＋並び替えを **AND重ねがけ**。地域マッピングは `REGION_MAP`。
  - コレクションは localStorage キー `wh_collection_v1`。
  - 地図は詳細を開いた時にLeaflet（CDN）を遅延ロード。
- `sw.js` … Service Worker（**cache `wh-v2`**。同一オリジンを成功時のみキャッシュ。**配信ファイルを更新したら必ず `CACHE` のバージョンを上げる**＝古いキャッシュ配信を防ぐ。検証時はSW登録解除＋caches.delete推奨）
- `manifest.webmanifest`, `icons/icon-192.png`, `icons/icon-512.png`

> パスはすべて相対。GitHub Pagesのサブパス（/world-heritage-app/）でそのまま動く。
